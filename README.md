# 概要。
各ソフトウェアに対する設定ファイルを保存する場所になる。  
設定ファイルが混在するため、見にくくなることだろう。  
それを本ファイルで説明することにより、少し緩和できたらと思う。

### 内容
バックアップファイルは、OS問わず、ホームディレクトリ直下の.vim\_backupディレクトリを使うようにしている。  
ただし、スワップファイルの作成は、カレントディレクトリに作るようにしている。  
その理由は言わずもがな。
ファイルサーバでの編集をする場合は、当然のようにカレントディレクトリが求められるからだ。  
しかし、ファイルの開き方によっては、スワップファイルがカレントディレクトリに作られないこともあるようだ(これも困る)。  


#### プラグイン無関係設定
日本語や多バイト言語の設定をしている。  
しかし、これは元からあった設定を残しているだけで、具体的な処理内容までは把握していない。  

C言語限定のデバッグ画面表示遷移処理は、どこかのホームページからコピペしただけで、どのように使うのか不明。  
標準のプラグインのため、外部から取り込まずに利用できるのはいい。  

vimgrep用にウィンドウが分割する設定をしている。  
タグなどの対となるキーワード移動も標準のプラグインを利用している。  

Windows版のvimは、香り屋さんのを使っている。  
そのための設定も(元から)入ったまま。  

バックアップファイルの作成箇所は1カ所に固めたいため、OS関係なくまとめる設定をしている。  
他で呟いているが、スワップファイルだけはカレントディレクトリに放置したままだ。  

文字コードの設定もどこかのホームページから真似したのだが、UTF-7やJISファイルは開けない。  
また、GUI用とCUI用で同じvimrcファイルを参照させているのだが、文字コードの解釈に違いが見られる(何とかしたい)。  

折りたたみ設定を有効化しているが、ファイルを開いた時点では、折りたたみを開いた状態にしている。  

* za	：トグル  
* zo	：折りたたみが開く  
* zc	：折りたたむ  
* zR	：すべての折りたたみを開く  
* zM	：すべての折りたたみを閉じる。  

秀丸エディタやサクラエディタのように、タブバーも使いやすいように設定している。  

* \<Leader\>tn	：新規タブを開く。  
* \<Leader\>tc	：タブを閉じる。  
* \<Leader\>tm	：カレントバッファを新規タブに移動する。  
* \<Leader\>to	：カレントタブ以外閉じる。  
* gvimrc
  * \<C-Tab\> gt	：Ctrl+Tabで、タブ移動(左から右)。  
  * \<S-C-Tab\> gT	：Ctrl+Tabで、タブ移動(右から左)。  

vimエディタの設定ファイルを開ける。  

* \<Leader\>ev	：vimrcを開く。  
* \<Leader\>eg	：gvimrcを開く。  
* Windows限定  
  * \<Leader\>eh	：AutoHotKeyを開く(ドキュメントディレクトリ配下配置が前提)。  

Windows限定として、
`<Leader>a`
をすることにより、カレントバッファの全選択ができるようにしている。  

gvimrc限定の設定としては、ウィンドウサイズを固定したままエディタ起動するとか、日本語や多バイト文字への対処などなど。  
如何せん。
日本語入力可能状態(インサートモード)からEscキー押下で抜けるときに、日本語入力をOFFにしてくれないプラグインがある(設定で対応可能だが、好みでない)。  

などなどの設定説明は、ざっくりではあるが、コメントを付けているため、混乱することはないはず。

#### プラグイン本体
OS問わず、プラグイン管理などは、ホームディレクトリ直下の.vimディレクトリを使うようにしている。  

k-takataさん作のminpacでプラグインを管理している。  

IDEに近づけるためのプラグインをいくつか導入し、コンパイル環境も用意している。
ただし、開発用のコンパイラが導入済みであることを前提にしているため、なかなか他の環境での実現は厳しいだろう(冒頭の目的を達成しにくい)。  
直近はGo言語を考えているが、なかなか技術が身につかないため、エディタ機能のマクロ自動記録を使うのがせいぜいだろう(vimのマクロとは言っていない)。  

環境によっては、プラグイン導入が不可能な場合があると思っている。  
その場合は、導入せずに運用できるよう、プラグイン未導入の場合の判断処理を入れている(if文での制御なので、たいしたことではないが、重要かな)。  
ただ、環境変数などの変更は放置したままだ(影響は無いと思っているので、起動時間の改善はないだろうが)。

##### プログラミング言語用プラグイン
今のところ以下を使っている。  

* vim-goimports  
プログラミング言語で利用するプラグインのひとつ。  
Go言語のインポート作業に必要な作業を自動でプラグインが担ってくれる。  
これがなければvimでプログラムを書こうと思わなかったぐらいだ。  
本当は、もっと他に優秀なのはあるが、多機能であり、手に余るようで、必要最小限のプラグインを選んだ次第だ。  
コンパイル環境が古い場合、動かない可能性もある。  

* vim-quickrun  
特定の言語用と言うより、プログラミングに必要なコンパイル環境と言えるプラグインだろう。  
ただし、前述通り、コンパイル環境があることを前提にしている。
また、コンパイル環境が古い場合、動かない可能性もある。  
不思議なのは、PowerShellを動かせないこと(微妙に困る)。
対応していないだけなので、不思議でも何でも無いけど。  

##### ctrlp.vim
ファイル検索プラグイン。  

有名なので導入したが、そもそも私は必要なファイルをディレクトリにあるファイルから直接Gvimエディタに投げて開くため、あまり使わないプラグインでもある。  

プログラミング規模が大きくなる、もしくはそれに準じたファイル数になれば重宝することだろう。  
私の場合は、同名ファイルの異なるPathを編集することが多いため、編集する場合は、わざわざPathをホームディレクトリからたどらなければ、違うファイルを変更してしまう心配を持っており、なかなか有効活用できていない。  

今後は、意識して活用しようと思う。  
せっかく導入したのだから・・・。  


##### Git用プラグイン
以下の2種類を組み合わせて使う人が多いようなので、私も真似をした。  

* vim-gitgutter  
* vim-fugitive  
* vim-rhubarb  

1つ目は、ファイル内の差分を視覚効果で表してくれるプラグインになる。  
2つ目は、Gitディレクトリ上のステータス確認に使うプラグインになる。  
3つ目は、2つ目のプラグインと連携し、ブラウザで開く。  

※vimエディタが古い場合、正常に動かないことがあるため、気をつけること。  


##### vim-showmarks
Windows使いであれば、秀丸エディタを活用していることだと思う。  
もしくは、サクラエディタだろうか。  
どちらにしても、現在行をマークやブックマークとして設定することができる。  

vimエディタでもできるのだが、なかなか視認しづらいと言うか、マーク箇所が全くわからないため、視覚できるようにしたプラグインになる。  
ただ、それでもマーク場所に移動しにくいことに変わりは無い。  
一覧で確認するならば、このプラグインを使う意味が無い。  
一応は直近のマークが記号列の部分に表示される。
そのため、上下に移動しやすくなる(当然直近だけに限るが)。  


##### gundo.vim
アンドゥ・リドゥをGitのような形で遡れる。  
Pythonを含んだコンパイル済みのvimエディタが必須になる。  

Python2.4以上を導入済みであること。  

##### undotree
アンドゥ・リドゥをGitのような形で遡れる。  
当たり前ではあるのだが、エディタを終了した場合、その履歴は消える。  
そして、バッファを切り替えただけでも履歴が消える(かなり困る)。  

##### 他には、、、
通常のIDEの機能として、Tabキーで補完候補を確定させるショートカットキーがある。  
それに倣ったプラグインを使おうと思った時期もある。  
しかし、通常のTabキーが無効化されるため、泣く泣く無効化したプラグインでもある。  

他には、Googleカレンダと連携できるプラグインもある。
